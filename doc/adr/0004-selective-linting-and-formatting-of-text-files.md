# 4. Selective linting and formatting of text files

Date: 2026-04-12

## Status

Accepted

Supercedes [3. lint and format text files](0003-lint-and-format-text-files.md)

## Context

While [ADR-0003](0003-lint-and-format-text-files.md) established that all
text-based files should be linted and formatted, some categories of files are
generated, auto-managed, or not meaningful to lint and format:

- Lock files (`.lock`) are generated by package managers and reflect transitive
  dependencies, not source code decisions
- Metadata files (`.metadata`) are generated by frameworks and tools
- Platform-specific generated directories (e.g., `ios/`, `android/`, `linux/`,
  `macos/`, `web/`, `windows/`) contain large amounts of generated boilerplate
  code and configuration

Attempting to lint and format these files produces noise, false violations, and
violates the principle of linting and formatting only meaningful source code.

## Decision

We lint and format only text-based files that represent meaningful source code or
configuration decisions.

The following file types and directories are implicitly excluded from linting
and formatting:

- Lock files (e.g., `*.lock`)
- Metadata files (e.g., `.metadata`)
- Generated platform directories: `ios/`, `android/`, `linux/`, `macos/`,
  `web/`, `windows/`

The `lefthook` configuration must be maintained to cover all meaningful text-based
file types that remain. When a new text-based file type is introduced, an
appropriate linter and formatter must be added as part of the same change.

## Consequences

The linting and formatting configuration becomes more focused on meaningful source
code, reducing false positives and noise in development workflows.

The `.gitignore` files in generated platform directories ensure that build
artifacts are not committed while the source configuration remains under version
control.

Contributors must understand which file types are excluded and why. Documentation
and exclusion patterns in `lefthook` must be maintained consistently.
